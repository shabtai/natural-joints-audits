Audit Findings

Finding 1 — Wrong model for “multiple with supervision” AAP predictions
BOTH VERIFIED NOVEL* MINOR
Rule typeModel-result mismatch (copy-paste error) Locationpraim/06_Table3.R:127, 219, 312, 404, 510, 611, 757, 872, 1000, 1117

In every “multiple with supervision” block across all 6 outcomes and all 4 cancer-subtype analyses, the Average Adjusted Predictions (AAP) are computed using avg_predictions(mod_sens2a, ...) — the model without supervision — while avg_comparisons correctly uses mod_sens3a (with supervision). The two halves of each result row come from different models.

### Multiple regression with supervision
cbind(regression = "multiple with supervision",
  avg_predictions(mod_sens2a, ...))   # WRONG: should be mod_sens3a
CDR_diff  <- avg_comparisons(mod_sens3a, ...)  # correct
CDR_ratio <- avg_comparisons(mod_sens3a, ...)

Impact: supervision=True accounts for 6.9% of rows (5.6% AI, 8.7% control). Cancer detection rate difference between supervised and unsupervised: <0.3%; recall ~5%. Predicted rates for this sensitivity row are modestly wrong.

Fix: Replace avg_predictions(mod_sens2a, ...) with avg_predictions(mod_sens3a, ...) in all “multiple with supervision” blocks.

Found by: blind-honesty, blind-code, enriched-honesty, enriched-code, enriched-facts.

Finding 2 — Resampling overwrite bug: all 1,000 iterations run on the original full dataset
BOTH VERIFIED NOVEL* MODERATE
Rule typeWrong data loading in resampling loop Locationpraim/Suppl_Resampling_InternalFunctions.R:13–14

Suppl_Resampling_InternalFunctions.R is source()d inside the 1,000-iteration loop in both resampling scripts. Its first two lines overwrite the resampled d1 that was just loaded:

# InternalFunctions.R lines 13-14:
load("Output/PRAIM_withWeights.RData")   # loads full dataset into d
d1 <- d                                  # overwrites the resampled d1!

# Calling loop (Suppl_Resampling_Subset.R:31-32):
load(file = paste0("Output/Subset/PRAIM_SubsetWithWeights_", s[j], ".RData"))
source("Code/Suppl_Resampling_InternalFunctions.R")  # immediately overwrites d1

All 1,000 iterations analyze the identical full dataset. The resampled CIs reflect only Monte Carlo estimator noise, not sampling variability.

Impact: Correct SE inflation factor = √(461,818 / 369,454) = 1.118. Reported resampling CIs are approximately 11.8% too narrow throughout. For results near significance thresholds this could shift conclusions.

Fix: Remove the load() and d1 <- d lines from InternalFunctions.R. The calling scripts already provide d1.

Found by: blind-code, enriched-code.

Finding 3 — Missing row comma: [-which(...)] performs column, not row, removal
BOTH VERIFIED NOVEL* NEGLIGIBLE
Rule typeWrong filter / boundary error Locationpraim/06_Table3.R:444, 547  |  praim/10_ExtDataTable3.R:272, 324

d_biopsy_mult <- d_biopsy_mult[-which(...)] is missing a trailing comma. In R, df[-vector] without a comma performs column subsetting, not row removal. The intended MISSING-value filter is silently broken for PPV multiple regression datasets.

Impact: 11 / 2,919 biopsy rows (0.38%) retain MISSING demographics. Negligible effect on coefficients.

Fix: d_biopsy_mult[-which(...), ] (add comma before ]).

Found by: blind-code, enriched-code, enriched-facts.

Finding 4 — na.omit(cols=) silently ignored for plain data.frames
BOTH VERIFIED NOVEL* NEGLIGIBLE (latent)
Rule typeSilent wrong filter Locationpraim/06_Table3.R:442, 546  |  praim/10_ExtDataTable3.R:270, 323

Base R’s na.omit.data.frame does not accept a cols argument. na.omit(d1, cols = c("correct_biopsy")) silently ignores cols and removes any row with any NA across all columns. Accidentally produces the correct result only because correct_biopsy is currently the only NA-carrying column. Fragile if upstream code introduces additional NAs.

Fix: d_biopsy_mult <- d_biopsy (already computed one line above).

Found by: blind-code, enriched-code, enriched-honesty, enriched-facts.

Finding 5 — stageX uses non-existent “STAGE_X” value; MISSING stage cases excluded from Table 3
BLIND+ENRICHED VERIFIED NOVEL* MINOR–MODERATE
Rule typeData coding inconsistency / non-existent category Locationpraim/06_Table3.R:907  vs  praim/10_ExtDataTable3.R:509

The main analysis encodes unknown stage as cancer_stage == "STAGE_X" | cancer_stage == "CANNOT_BE_DETERMINED". “STAGE_X” does not exist in the data. The sensitivity analysis (ExtDataTable3) correctly uses "MISSING" | "CANNOT_BE_DETERMINED". As a result, 117 detected cancers with cancer_stage == "MISSING" (4.1% of all detected cancers) receive all-zero stage indicators in Table 3, silently treated as reference-category observations. A dead variable stageU is initialized but never assigned (incomplete edit).

Impact: 117 / 2,881 detected cancers (4.1%) miscategorized. Given only 56 Stage 3/4 cancers in the sample, absorbing 117 misclassified cases substantially distorts Stage 3/4 rate estimates.

Enrichment contributed: PRAIM.yaml enumerated all 8 valid cancer_stage values, revealing “STAGE_X” is absent.

Fix: d1$stageX[which(d1$cancer_stage %in% c("CANNOT_BE_DETERMINED","MISSING"))] <- 1 in both files.

Found by: blind-honesty, blind-code, enriched-honesty, enriched-code, enriched-facts.

Finding 7 — sn_acc_cancer labeled “cancers saved by AI” is a co-occurrence measure, not a counterfactual
BLIND VERIFIED NOVEL* MAJOR
Rule typeData misrepresentation / causal overclaim Locationpraim/05_Table2.R:13–16

The variable sn_acc_cancer is defined as cancer_detected == "True" AND safety_net_accepted == "True" with the comment: “these cancers wouldn’t be found without AI.” This is a co-occurrence indicator. It provides no causal evidence that the cancer would have been missed without the safety net.

# 05_Table2.R lines 13-16 (comment is the authors' own):
# (these cancers wouldn't be found without AI)
d$sn_acc_cancer <- "False"
d$sn_acc_cancer[which(d$cancer_detected == "True" &
                      d$safety_net_accepted == "True")] <- "True"

Impact: In the example data, every one of the 204 cases meeting this definition had already been flagged as suspicious by at least one human reader independently. The count of cases where the safety net was the sole mechanism preventing a miss appears to be 0 in the example data. The headline count of “AI-saved cancers” is substantially inflated.

Found by: blind-honesty only. Track B auditors found related Table 5 issues but not this specific claim.

Finding 8 — Cancer subtype regressions use all-screened-women denominator, not cancer-cases-only
BLIND VERIFIED NOVEL* MINOR (framing)
Rule typeClinical intent gap / denominator conflation Locationpraim/06_Table3.R ~lines 677–1130 (grade, stage, size, invasiveness)

Binary indicators such as d1$G2 <- ifelse(invasive_grade == "G2", 1, 0) are created over all rows, including non-cancer rows where invasive_grade = "NOT_APPLICABLE" (coded 0). The regression measures the detection rate of G2 cancers per screen, not the proportion of G2 among detected cancers. A code comment acknowledges this design choice. Both framings are internally valid but produce 4.2× different numbers (G2: 109.6 per 1,000 screens vs. 45.9% of cancers) and invite misinterpretation.

Found by: blind-honesty only. Not flagged by enriched track.

Finding 9 — ai_prediction used as confounder but appears as mediator in authors’ own DAG
BLIND PARTIALLY VERIFIED NOVEL* MINOR
Rule typeDAG specification question Locationpraim/03_Weights.R:12  |  praim/14_ExtDataFigure6.R

The DAG in 14_ExtDataFigure6.R codes ai_prediction -> used_ai_viewer -> cancer_detected, placing ai_prediction on a causal path from background disease to treatment. adjustmentSets() on this DAG does return {ai_prediction, reader_set} as a valid adjustment set, so the authors’ own structural model internally justifies the choice. The concern remains valid if ai_prediction is generated simultaneously with — rather than strictly before — the decision to use the AI viewer.

Impact: Cramer’s V (ai_prediction vs used_ai_viewer) = 0.028 in example data. Practical impact on PS estimates is minor regardless of interpretation.

Found by: blind-honesty only.

Finding 10 — Table 5 counterfactual: PS weights computed before outcome zeroing
BLIND+ENRICHED VERIFIED NOVEL* MINOR (conceptual)
Rule typeWeights–estimand mismatch in counterfactual analysis Locationpraim/08_Table5.R:30

Table 5 models a hypothetical protocol where AI-tagged “normal” studies bypass radiologist review (outcomes zeroed). The zeroing is applied after loading PS weights computed on real, unmodified data. The weights calibrate the real-world population distribution; they do not correspond to the counterfactual population with zeroed outcomes.

Enriched track additionally noted that zeroing also applies to cases where safety_net_triggered == True, potentially over-attributing the counterfactual effect.

Found by: blind-honesty, enriched-honesty.

Finding 11 — screening_unit excluded from all PS models despite being the dominant confounder
ENRICHED VERIFIED NOVEL* MAJOR
Rule typeOmitted variable bias / confounding Locationpraim/03_Weights.R:12 (all PS models)

The PS model uses only used_ai_viewer ~ ai_prediction + reader_set. screening_unit (12 sites) is never included in any PS model, sensitivity analysis, or cluster correction — despite being the strongest predictor of AI viewer use in the data.

  • Cramer’s V (screening_unit vs used_ai_viewer) = 0.533
  • screening_unit_12: 0% AI use (149 studies)
  • screening_unit_07: 96.5% AI use (1,612 studies)
  • screening_unit_03: 89.5% AI use (1,173 studies)
  • Cancer detection rate range across units: 18.95%–29.53% (SD = 3.67 pp)
  • 24.3% of sample rows are in the three extreme-allocation units

Screening unit is a near-perfect proxy for AI assignment in several sites and independently associated with outcomes. Excluding it leaves major structural confounding uncontrolled in all primary and sensitivity analyses.

Enrichment contributed: semantic layer documented screening_unit as a 12-category site identifier with known AI deployment variation, directly prompting this check.

Found by: enriched-honesty only. Not found by blind track.

Finding 12 — Subgroup analyses apply full-cohort PS weights without re-estimation
ENRICHED VERIFIED NOVEL* MINOR–MODERATE
Rule typeCausal inference design / subgroup weight misalignment Locationpraim/07_Table4_ExtDataTable2.R

Table 4 subgroup analyses (by screening round, breast density, age, hardware vendor) subset the data and apply the full-cohort PS overlap weights directly — no re-estimation of propensity scores within each subgroup. Overlap weights computed on the full cohort balance the marginal distribution but are not guaranteed to balance within subgroups where the PS distribution may differ.

Found by: enriched-honesty only. Not found by blind track.

Finding 14 — Safety-net alert displayed in only ~13% of triggered cases
BLIND+ENRICHED VERIFIED NOVEL* MINOR (descriptive)
Rule typeData honesty / clinical context gap Locationpraim/05_Table2.R

In example CSV: safety_net_triggered=True in 1,975 rows; safety_net_shown=True in only 260 of those (~13%). The safety net was generated internally but withheld from radiologists in ~87% of triggered cases. Any analysis framing the safety net as a clinical intervention should foreground this distinction.

Enrichment contributed: documented the three-step cascade (triggered → shown → accepted) and the logical dependency that shown requires used_ai_viewer=True.

Found by: blind-honesty, enriched-facts.

Enrichment Value: HIGH

HIGH — Track B found 3 unique findings (F11, F12, F15) that Track A missed entirely:

  • F11 (MAJOR) — screening_unit excluded from PS. Cramer’s V = 0.53 between site and treatment. Only identified because the semantic layer documented screening_unit as a site-level variable with known AI deployment variation.
  • F12 (MINOR–MODERATE) — Full-cohort PS weights applied to subgroups. Enabled by enrichment’s grain analysis.
  • F15 (LOW) — target overwrite in supplementary stratification script.

Enrichment additionally deepened F5 (stage value enumeration) and F14 (safety-net cascade logic).

Key Takeaways

  • 13 verified, 1 partially verified, 1 refuted out of 15 total findings.
  • 2 MAJOR findings: F7 (safety-net “saves” are co-occurrence, not causal) and F11 (screening_unit excluded from PS; Cramer’s V = 0.53).
  • 1 CRITICAL code bug: F2 (resampling overwrite) — all 1,000 resampling iterations analyze identical full data; reported CIs are 11.8% too narrow.
  • Top finding by impact: F11 — the strongest predictor of AI viewer use (screening_unit, V = 0.53) is entirely absent from all propensity score models, leaving site-level confounding uncontrolled in all primary estimates.
  • Second-highest impact: F7 — the headline count of “cancers saved exclusively by AI” appears to be near zero when the co-occurrence definition is interrogated against the data.
  • Most prevalent code class: Silent wrong filters (F3, F4) and copy-paste errors (F1) — easy to fix but found in 4+ locations each.
