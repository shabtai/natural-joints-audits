Audit by Natural Joints
Headline: no published claim of this paper is affected. An audit of the released model code, configs, and the public datasets the paper validates on (the GSE185263 sepsis cohort and the GSE148073 type-1-diabetes example) found no data-integrity error, no fabrication, and no result-changing defect. Every number we could recompute from the released artifacts reproduces exactly — the sepsis cohort construction, the figure cohort counts, the 18,301-gene vocabulary, the model architecture, and the reported parameter counts. The paper is also unusually candid about its own limits (it openly reports a sepsis AUROC of 0.56 and that a trivial PCA+XGBoost baseline ties it on classification).

The only issues we surfaced are four minor defects in the released code — reuse hazards that would bite a third party adapting the published repositories to new data, but that have no effect on any result reported in the paper. We report them here for the authors and for anyone reusing the code.
Scope. This audit covers the released code and the public datasets in the paper’s validation set. The headline drug-target rankings (lupus / multiple myeloma) depend on processed atlases and a ranking/curation pipeline that are not yet public (“available upon publication”), which is normal for a preprint; those numbers were therefore not independently re-derived and are out of scope here, not in dispute.

Released-code notes (do not affect the paper’s results)

Four real, verified defects in the published code. Each is a hazard only for someone reusing the repositories on their own data; none changes a number reported in the paper. We confirmed this for each.

Note 1 — Evaluation drops the patient grouping, so checkpoint selection uses the wrong loss aggregation
VERIFIED NOVEL NO PAPER IMPACT
Wherevirtual-cell-patient/train.py (prediction_step; metric_for_best_model="eval_loss") Paper impactNone under the documented augmentation

Training averages the loss per patient (to stop patients with more augmented views from dominating), but the evaluation path pops entity_id before the forward pass, so the validation loss — which selects the best checkpoint and drives early stopping — falls back to a plain per-view average. Model selection is therefore biased toward patients with more views, the exact thing the per-patient averaging was meant to prevent.

Why it doesn’t affect the paper. When every patient has the same number of augmented views, the per-view and per-patient losses are mathematically identical, so the bug is inert. The released example uses a uniform 5 views per patient and the README documents 5× as the default — under that scheme checkpoint selection is unchanged. The bug only bites if view counts vary across patients. (Training gradients are correct either way; only checkpoint selection could shift.)

Note 2 — Patient IDs are round-tripped through float32 and collide above 224
VERIFIED NOVEL NO PAPER IMPACT
Wherevirtual-cell-patient/train.py (entity_id packed into the logits tensor, cast back with .astype(int)) Paper impactNone (4,479 patients « the collision boundary)

To carry patient identity into the metric function, entity_id is appended to the logits as a float column and later cast back to int. float32 represents integers exactly only up to 224 = 16,777,216; above that, distinct patient IDs silently merge (reproduced: 16,777,216 and 16,777,217 collapse to the same value). Merged IDs would corrupt the per-patient metric aggregation.

Why it doesn’t affect the paper. The training corpus is 4,479 patients — far below the boundary — so no collision is possible in the paper’s own run. The hazard is for a re-user who assigns large or hashed integer patient IDs.

Note 3 — README quick-start feeds un-normalized random input to a normalization-sensitive model
VERIFIED NOVEL NO PAPER IMPACT
Wherevirtual-cell-patient/README.md quick-start (torch.randn(1, 500, 18301)) Paper impactNone — not part of the paper’s pipeline

The model expects non-negative, library-size-normalized, log1p-transformed expression. The quick-start example feeds Gaussian noise (which contains negatives) and, because it runs under .eval(), “succeeds” and returns a class — so a user could mistake the shape-only smoke test for a valid inference recipe and run the model on out-of-distribution input.

Why it doesn’t affect the paper. The snippet is documentation, not part of the paper’s analysis; the paper’s own inputs are correctly normalized (TPM / CP10k+log1p). The fix is to ship a normalized example or label the snippet as shape-only.

Note 4 — BatchNorm requires batch > 1, but the trailing batch isn’t dropped
PARTIALLY VERIFIED NOVEL NO PAPER IMPACT
Wheremodeling_virtual_cell.py (assert x.shape[0] > 1); train.py (dataloader_drop_last unset) Paper impactNone — a crash would have prevented training

The encoder asserts a batch size greater than 1 (BatchNorm needs ≥2 samples), yet dataloader_drop_last is not set. A dataset whose size leaves a size-1 trailing batch can therefore crash mid-epoch — most relevant to the distilled bulk model, whose input has no cell dimension to pad the batch out.

Why it doesn’t affect the paper. A crash would have stopped training outright; the released checkpoints trained fine, so this did not occur for the authors’ batch sizes. It is a latent crash risk for a re-user. The fix is one line: dataloader_drop_last=True.

Reading guide

This report is a negative result: a rigorous audit of a paper that holds up. The released data is authentic and integrity-clean, every checkable number reproduces exactly, and the authors are candid about the genuine limits of their results. The four items above are ordinary code-quality defects in the released repositories — useful to fix for reproducibility and reuse, but with no bearing on any conclusion the paper draws.

Items considered and deliberately not elevated: the full training corpus, evaluation cohorts, and ranking/curation code are not yet public (“available upon publication”), which is the normal state of a preprint and not a defect; and a few abstract/figure wording choices are expanded transparently in the paper’s own body and supplement. Neither category constitutes a claim-invalidating finding, so neither is treated as one here.
